# Quality Comparison: Qwen 3.6-35B-A3B (Local) vs. Kiro (Cloud) — Local LLM Deployment Reports

> Side-by-side quality evaluation of two reports on the same topic (Local LLM Deployment across Windows, macOS, and Linux), generated by different systems:
> - **Report A**: `local-llm/report-generated-by-qwen36-a3b.md` — Generated by a local Qwen 3.6-35B-A3B model via a LangGraph multi-agent pipeline (researcher → security reviewer → performance reviewer → aggregator)
> - **Report B**: `reports/local_llm_deployment_report.md` — Generated by Kiro using web search, source fetching, and synthesis

---

## Overall Assessment

Both reports are technically substantive and cover the same core territory: OS architecture implications for LLM inference, kernel tuning, security hardening, performance benchmarks, and deployment recommendations. They reach the same strategic conclusion (Linux > Windows > macOS for production throughput). However, they differ significantly in factual grounding, currency, and verifiability.

| Dimension | Report A (Qwen Local) | Report B (Kiro) |
|---|---|---|
| Factual accuracy | Mixed — contains fabricated specifics alongside correct claims | Good — claims are sourced and verifiable |
| Currency | Outdated (no MLX coverage, pre-Ollama 0.19) | Current (MLX, Ollama 0.19+, M4 chips, 2025–2026 toolchain) |
| Depth | Very deep — granular compiler flags, batch-size matrices, OS-specific memory management | Deep — covers the same topics with less granularity but more accuracy |
| Verifiability | Low — no sources cited; benchmark numbers appear model-generated | High — 9 external sources cited with URLs; benchmark data traceable |
| Actionability | High — copy-paste commands for all three OSes | High — copy-paste commands, though slightly less exhaustive |
| Balance | Good — acknowledges trade-offs for each OS | Good — similar balance with slightly more nuanced macOS coverage |
| Structure | Clear 5-section structure; heavy use of tables | Clear 9-section structure; better separation of concerns |

## 1. Factual Accuracy

### Report A (Qwen) — Significant Issues

Report A's biggest weakness is that many specific numbers appear to be hallucinated or fabricated by the local model. There is no way to verify them because no sources are cited.

- **Issue (Major)**: The Windows benchmark table claims a 7B Q4_K_M model achieves "180–210 tok/s" on an RTX 4090 at batch=1, context=4K. This is implausibly high. Real-world community benchmarks show RTX 4090 achieving ~95–130 tok/s for 7B Q4 models on Linux, and Windows would be 10–30% slower. The 180–210 figure appears fabricated.
- **Issue (Major)**: The Linux benchmark table claims "38–45 tok/s" for a 7B Q4_K_M on RTX 4090 at batch=1, context=4K. This is implausibly *low* — it contradicts the Windows figure in the same report (which claims 180–210 for the same model on the same GPU) and contradicts real benchmarks (~95–130 tok/s). The Linux and Windows numbers are internally inconsistent.
- **Issue (Major)**: The macOS section references only M2/M3 chips with bandwidth up to 400 GB/s. There is no mention of M4 chips (M4 Pro: 273 GB/s, M4 Max: 546 GB/s) or the MLX framework, which is the single most important development in the Apple Silicon LLM ecosystem since 2025. This is a critical omission that dates the report.
- **Issue (Major)**: The macOS section states "vLLM/CUDA entirely unsupported" — this is correct, but fails to mention `vllm-mlx`, an experimental port that enables vLLM-style serving on Apple Silicon via MLX.
- **Issue (Minor)**: Claims "Q4_K_M sweet spot avoids MPS precision instability" — while Q4_K_M is indeed more stable than Q2/Q3 on MPS, the framing implies MPS is the primary macOS backend. In 2025–2026, MLX has largely replaced MPS for inference workloads.
- **Issue (Minor)**: The `hw.cpumaxratio=100` sysctl for "pinning to P-cores" is not a documented macOS sysctl. This appears to be fabricated.
- **Issue (Minor)**: States `vm.overcommit_memory=1` in the performance section but then contradicts this with `vm.overcommit_memory=0` in the security section, with a note to "revert to 0." This internal contradiction is confusing.
- **Issue (Nit)**: The bandwidth utilization table claims "450–650 GB/s (eff)" for Q4_K_M on a 7B model. This figure doesn't correspond to any specific GPU's memory bandwidth and lacks context (which GPU? which bus?).

### Report B (Kiro) — Minor Issues

- **Issue (Minor)**: The WDDM performance penalty is stated as "10–30%" which is a wide range. The lower end (10%) applies to simple single-stream inference; the upper end (30%) applies to workloads with many small kernel launches. The range could be more precisely scoped.
- **Issue (Minor)**: The A100 benchmark row shows "~150+ tok/s" for 7B Q4 which is plausible but not directly sourced — the cited benchmarks focus on consumer GPUs.
- **Issue (Nit)**: States Linux kernel "6.x/7.x" in the scope line. Kernel 7.x is speculative as of the report date (kernel versioning is arbitrary and Torvalds hasn't announced 7.0).

## 3. Depth and Technical Detail

Report A wins on raw volume of technical detail, particularly in areas where the local model could draw on its training data:

- **Compiler flags**: Report A provides specific MSVC, NVCC, and GCC flags for building llama.cpp. Report B mentions these topics but doesn't enumerate specific flags.
- **Batch-size performance matrices**: Report A provides detailed TPS breakdowns by batch size and context length for each OS. Report B provides simpler per-GPU tables without batch-size variation.
- **Memory management internals**: Report A discusses `VirtualAlloc`, `MTLBuffer` storage modes, `cudaMallocHost()` internals, and `posix_fadvise` hints. Report B covers the same concepts at a higher level.
- **Windows registry tuning**: Report A provides more registry keys (pagefile min/max, NUMA policy, NVMe write cache, interrupt affinity). Report B covers the essential three (TDR, LargeSystemCache, NVMe interrupt moderation).

However, much of Report A's additional detail is unverifiable and some is demonstrably incorrect (see Section 1), which undermines the value of the extra depth.

Report B compensates with better structural organization — separating benchmarks, ecosystem comparison, known issues, and deployment recommendations into distinct sections rather than mixing them together.

## 4. Security Coverage

Both reports cover the same attack vectors (API exposure, model poisoning, prompt injection, memory leakage) and provide copy-paste hardening commands for all three OSes.

| Aspect | Report A (Qwen) | Report B (Kiro) |
|---|---|---|
| Attack vector analysis | More detailed exploitation paths (DNS tunneling, C2 beaconing, NaN gradient weaponization) | More grounded — cites real research (35% of Ollama instances unauthenticated, OWASP Top 10) |
| Windows hardening | More comprehensive (outbound firewall rules, ACL scripts, model hash verification script, WSL shutdown) | Covers essentials (localhost binding, firewall, pagefile encryption, Defender exclusions) |
| macOS hardening | Includes IP forwarding disable, verification checklist | Similar scope; cleaner pf.conf example |
| Linux hardening | Includes Docker security flags, LUKS swap encryption, cgroup tuning | Similar scope; includes `ollama/ollama` Docker image reference |
| Supply chain risk | Mentioned in attack vector table but no mitigation beyond ACLs | Explicitly calls out "no built-in signature verification in most tools" |

Report A's security section is more exhaustive but includes some speculative attack scenarios (e.g., "MPS casting instability weaponizable") that lack real-world evidence. Report B's security section is more grounded in documented vulnerabilities.

## 5. Verifiability and Sources

This is Report B's strongest advantage and Report A's most critical weakness.

- **Report A**: Zero sources cited. Every benchmark number, every performance claim, every technical specification is presented without attribution. Given that several numbers are demonstrably incorrect or internally inconsistent, the lack of sources makes it impossible for a reader to distinguish accurate claims from hallucinations.
- **Report B**: Nine external sources cited with full URLs, covering benchmarks (singhajit.com, Starmorph), OS-level analysis (eunomia.dev), security research (OWASP, arxiv), ecosystem guides (Red Hat, FOSS Linux), and hardware guides (Introl). Benchmark data is traceable to specific community measurements.

### Verdict

**Report B (Kiro) is the more reliable and useful reference document.** While Report A impresses with its volume of technical detail, the lack of sources combined with demonstrably fabricated benchmark numbers (180–210 tok/s for a 7B model on Windows, internally inconsistent Linux figures) means a reader cannot trust any specific claim without independent verification. Report A also misses the most significant development in the Apple Silicon LLM ecosystem (MLX), which dates it significantly.

Report B trades some granularity for accuracy and verifiability — every major claim can be traced to a cited source, the benchmark data aligns with community measurements, and the coverage reflects the current state of the toolchain. For a reference document that an engineer would actually rely on for deployment decisions, verifiability matters more than volume.

**Report A's strength** is its depth on low-level optimization (compiler flags, batch-size performance regimes, memory management internals). If its claims could be independently verified and its macOS coverage updated, it would be the stronger document.

**Report B's strength** is that you can trust what it says. In a domain where incorrect kernel tuning or fabricated benchmark expectations can lead to real operational failures, that matters.
